# Modeling Language Paradigms

## Item-and-Process Morphology in Pynini
### Introduction
In computational morphology, we face a choice between two primary theoretical frameworks for word formation: **item-and-arrangement**, which treats words as simple concatenations of stems and affixes, and **item-and-process**, which views affixation as a sequence of transformational processes applied to a base
. Finite-state text processing strongly favors the item-and-process model. As Gorman and Sproat [GormanFinite] note, such processes are computationally equivalent to rational relations and can be modeled elegantly by cascading finite-state transducers (FSTs).

To implement this, the Pynini library provides the pynini.lib.features and pynini.lib.paradigms modules. These tools allow the grammar engineer to define abstract morphosyntactic properties, map them to affixation transducers, apply context-dependent phonological rules, and lazily compile bidirectional analyzers and generators. 

Throughout this chapter, we will ground these computational concepts using the Irish language as our primary example. Irish is an excellent candidate for item-and-process modeling because it requires suffixes to harmonize with the stem—a rule known as leathan le leathan, caol le caol (broad with broad, slender with slender).

## Defining Morphosyntactic Features (pynini.lib.features) 
Before a system can build word forms, it must understand the abstract grammatical space—the features that define a word's place in a paradigm (e.g., its tense, person, or case).

The Pynini library has a dedicated module for working with such paradigms that represent linguistic feature spaces. The paradigms is an abstraction that applies to all Parts of Speech in a language. IN the example below we illustrate it with an application to the **Verb phrase**.

### Instantiating Individual Features
We begin by defining individual morphological categories using the features. ***Feature class***, specifying the feature's name and its valid values. In the Irish regular verb system, verbs conjugate for properties like Tense and Person. These properties are specific to the language and are not universal. Each language will potentially have a different set of features.


```python
from pynini.lib import features, paradigms

tense = features.Feature("tense", "pres", "fut", "past")
person = features.Feature("person", "1", "2", "3")
number = features.Feature("number", "sg", "pl")
```

### Bundling into a Category
Individual features are bundled into a features.Category object. The Category acts as an acceptor that validates any sequence of feature-value pairs it contains.
```python
verb = features.Category(tense, person, number)
```

### Defining Feature Vectors
We use features:  ***FeatureVector*** to represent a valid, specific combination of a Category and its values, representing a specific slot in the paradigm. We must also define a "lemma" feature vector, which acts as the citation form of the word.

```python
# The 1st person plural present (e.g., "we break")
pres_1pl = features.FeatureVector(verb, "tense=pres", "person=1", "number=pl")

# The citation form (lemma)
lemma_features = features.FeatureVector(verb, "tense=pres", "person=3", "number=sg")
```

## Modeling Morphotactics & Affixation (pynini.lib.paradigms)
Once the feature space is established, we map feature vectors to the actual FST operations that construct the word forms. In the paradigms module, affixes are introduced via FST composition rather than raw concatenation, allowing the affix to impose constraints or trigger changes on the stem.

### Defining the Stem Shape
We select a boundary symbol (conventionally +) to separate stems from affixes and define the baseline stem acceptor using paradigms.make_byte_star_except_boundary().

```python
stem = paradigms.make_byte_star_except_boundary()
```
### Constructing Slots with Archiphonemes
A paradigm is defined by a list of slots, where each slot is a Python tuple pairing an affixation transducer (like paradigms.suffix()) with a FeatureVector.

The Irish Example: In Irish, the present tense suffix relies on the broad/slender quality of the stem. For a broad stem like _cas_ (twist), the 3rd person present suffix is -ann (_casann_). For a slender stem like bris (break), the suffix is -eann (_briseann_). So note that the 'function' of the suffix is the same; 3rd person present suffix, but the surface form changes based on the context of the word.

To encode this without hardcoding two separate paradigms, we use an abstract "**archiphoneme**" (e.g., A) in our suffix definitions. These archiphonemes will be resolved into surface forms later in the pipeline.

>Note: the use of such an "archiphoneme" is a common practice in computational morphology to handle allomorphy without explicitly listing every possible surface form. We basically create an intermediate form representing the allomorphs of a morpheme. Based on the context of the word (which we will define later) the archiphoneme will be rewritten to the correct surface form of the morpheme.

```python
slots = [
    # 3rd Person Present: cas+Ann → casann, bris+Ann → briseann
    (paradigms.suffix("+Ann", stem), 
     features.FeatureVector(verb, "tense=pres", "person=3")),
     
    # 1st Person Plural: cas+Aimid → casaimid, bris+Aimid → brisimid
    (paradigms.suffix("+Aimid", stem), pres_1pl),
]
```

### Mapping the Grammatical Space

In a robust computational grammar, the `slots = []` collection is far more than a simple list of suffixes; it represents the **full set of grammatical space cells** defined by your `Category`. Every unique combination of features (e.g., Tense × Person × Number) that results in a distinct morphological or phonological realization must be instantiated as a slot.

As the grammar grows, this collection expands to cover the entire cross-product of feature values:

1. **Cell Instantiation**: Each `FeatureVector` represents a single point (or a subset of points) in the multi-dimensional grammatical space.
2. **FST Mapping**: For every cell, you provide an FST operation (like `paradigms.suffix` or `paradigms.prefix`). 
3. **The Logical Union**: Internally, Pynini treats this list as a union of relations. When you call `.inflect()`, the library identifies which `FeatureVector` matches your request and applies the corresponding FST operation.

By defining a slot for every possible grammatical intersection, you ensure that the transducer can deterministically map any abstract linguistic request to its specific "logical shape" (the archiphoneme string) before the phonological rules resolve it to the surface form.

### The Scale of the Grammatical Space

In Irish, the magnitude of the verbal grammatical space is a product of its synthetic richness. A single regular verb lexeme typically maps onto a multi-dimensional matrix defined by:

*   **Tense-Aspect-Mood (TAM)**: 8 primary categories (Present, Past, Future, Imperfect/Past Habitual, Conditional, Present Subjunctive, Past Subjunctive, and Imperative).
*   **Person and Number**: 7 distinct roles per TAM category (1sg, 2sg, 3sg, 1pl, 2pl, 3pl, and the Impersonal/Autonomous form).
*   **Syntactic Dependency**: 2 statuses (Independent vs. Dependent), which distinguish forms used in simple assertions from those used after particles like *ní* (not), *an* (interrogative), or *go* (that).

This results in a core verbal matrix of approximately **112 unique cells** ($8 \times 7 \times 2$) for a single verb. When considering initial mutations (Lenition, Eclipsis, etc.) as additional axes in the transduction pipeline, the total number of "logical slots" the system must account for can easily exceed 400 per major part of speech. Modeling this density requires the modularity of the `slots = []` approach, where each cell is a specific instantiation of the linguistic theory.

## Integrating Morphophonological Rules
Because we used the **abstract A** in our affixation step, we must integrate a morphophonological rule to resolve it into surface orthography. We do this using pynini.cdrewrite(), which compiles context-dependent rewrite rules into FSTs. 

The Irish Example: Vowels a, o, u are broad, while e, i are slender. A consonant is broad if preceded by a broad vowel, and slender if preceded by a slender vowel. Thus, A should rewrite to a after broad sequences, and e (or i) after slender sequences.

```python
import pynini

broad_v = pynini.union("a", "o", "u", "á", "ó", "ú")
slender_v = pynini.union("e", "i", "é", "í")
cons = pynini.union("b", "c", "d", "f", "g", "h", "l", "m", "n", "p", "r", "s", "t")

# Define the alphabet closure
sigma_star = pynini.union(broad_v, slender_v, cons, "A").closure().optimize()

# Rule 1: 'A' → 'a' if preceded by a broad vowel + consonants
rule_broad = pynini.cdrewrite(
    pynini.cross("A", "a"), broad_v + cons.plus, "", sigma_star)

# Rule 2: 'A' → 'e' if preceded by a slender vowel + consonants
rule_slender = pynini.cdrewrite(
    pynini.cross("A", "e"), slender_v + cons.plus, "", sigma_star)

# Cascade the rules
vowel_harmony = (rule_broad @ rule_slender).optimize()
```

By separating the morphotactics (the slots) from the phonology (the 'pynini.cdrewrite rules'), we maintain the strict modularity demanded by the item-and-process model.


## Compiling the Paradigm (paradigms.Paradigm)
The Paradigm class brings together the syntactic features, morphological slots, lexicon, and phonological rules into a single object. When we instantiate the paradigm, we pass our `vowel_harmony FST` to the rules= argument. Pynini will automatically compose this rule with the affixation step.

```python
# first conjugation verbs

first_conj_verbs = paradigms.Paradigm(
    category=verb,
    name="1st Conjugation Regular Verbs",
    slots=slots,
    lemma_feature_vector=lemma_features,
    stems=["cas", "bris", "glan"], # Irish stems
    rules=[vowel_harmony]          # Automatically apply 'leathan le leathan'
)
```

## Utilizing the Compiled Transducers (Lazy Generation)
A major advantage of using Pynini's Paradigm class is lazy transducer generation. Because FSTs are mathematically invertible, the developer does not need to write separate code for parsing and generation. Instead, the single Paradigm definition dynamically constructs directional FSTs when called.

### Generation (Inflection)
To generate a surface word from abstract features, we access the .inflector property, which maps a feature string (e.g., _cas_ [tense=pres] [person=1] [number=pl]) to its surface form. Pynini automatically applies the vowel_harmony rewrite rule during composition.

```python
from pynini.lib import rewrite

# Outputs: "casaimid"
surface_form = rewrite.one_top_rewrite(
    "cas [tense=pres] [person=1] [number=pl]", 
    first_conj_verbs.inflector
)
```

### Analysis (Parsing)
Conversely, accessing the .analyzer property mathematically inverts the relations. It maps a surface string back to its lemma and morphosyntactic features, perfectly reversing the vowel harmony rule and affixation.

```python
# Outputs: "bris [tense=pres] [person=3] [number=sg]"
analysis = rewrite.one_top_rewrite(
    "briseann", 
    first_conj_verbs.analyzer
)
```

Through this architecture, the computational linguist achieves an elegant, highly modular grammar capable of robust generation and parsing from a unified set of linguistic descriptions.

## Advanced Construction: Mapping the Grammar Space Efficiently

A critical question arises when modeling complex parts of speech: **Do we need to explicitly fill every single cell in the grammatical space?**

### Defaults vs. Explicit Overrides
In Pynini's `paradigms.Paradigm`, the answer is **no**. You do not need to list every possible feature permutation if they share the same morphological realization. We use two strategies to minimize repetition:

1.  **Feature Underspecification (The "Default" Logic)**: A `FeatureVector` that only specifies a subset of features (e.g., `[tense=pres]`) will match *all* slots that share that value, regardless of their person or number. This effectively assigns a "default FST" to a broad region of the grammar space.
2.  **Explicit Overrides**: Pynini's inflector logic is deterministic. By placing more specific feature vectors (e.g., `[tense=pres, person=P1, number=pl]`) after or in conjunction with broader ones, the system can provide specialized "overrides" for the irregular or "odd" cells (the allomorphs) while leaving the rest to the default.

By carefully layering these definitions, a grammar engineer can define a 112-cell verbal paradigm with only a dozen or so `slots` entries.

### Implementing the Multi-Cell Map
In the Pynini `slots` list, this matrix is implemented by grouping identical realizations (underspecification) and overriding unique ones.

```python
# Blueprint Implementation (1st Conjugation Fragments)
slots = [
    # 1. Broad Defaults for Present (3rd Person/Analytic)
    (paradigms.suffix("+Ann", stem), 
     features.FeatureVector(verb, "tense=pres")),
     
    # 2. Overrides for Synthetic Forms
    (paradigms.suffix("+Im", stem), 
     features.FeatureVector(verb, "tense=pres", "person=P1", "number=sg")),
     
    (paradigms.suffix("+Aimid", stem), 
     features.FeatureVector(verb, "tense=pres", "person=P1", "number=pl")),

    # 3. Autonomous (Impersonal)
    (paradigms.suffix("+Atar", stem), 
     features.FeatureVector(verb, "tense=pres", "person=Pauto")),
]
```

By mapping the "logical intent" (the features) to these archiphoneme instructions, the analyst ensures that the subsequent **Orthographic Layer** can perform a single, global pass of Vowel Harmony to produce the correct surface spelling for *every* cell without needing to hardcode 112 individual strings.

### Realizing the Surface Form: The Rule Cascade

The archiphonemes introduced in the `slots` list (e.g., `+Ann`, `+Im`, `+Atar`) are not the final surface forms. They are instructions for a follow-up **Morphophonological Rule Cascade**. In Pynini, this is achieved through a sequence of `cdrewrite` operations that resolve the abstract markers based on the stem's phonological environment.

The following rules represent the final processing stage that brings the word to the surface:

```python
# 1. Resolve 'A' (Broad → a, Slender → ea)
rule_A = (
    pynini.cdrewrite(pynini.cross("A", "a"), broad_v + cons.plus, "", sigma_star) @
    pynini.cdrewrite(pynini.cross("A", "ea"), slender_v + cons.plus, "", sigma_star)
)

# 2. Resolve 'I' (Broad → ai, Slender → i)
rule_I = (
    pynini.cdrewrite(pynini.cross("I", "ai"), broad_v + cons.plus, "", sigma_star) @
    pynini.cdrewrite(pynini.cross("I", "i"), slender_v + cons.plus, "", sigma_star)
)

# 3. Resolve 'Á' (Broad → á, Slender → é)
rule_Á = (
    pynini.cdrewrite(pynini.cross("Á", "á"), broad_v + cons.plus, "", sigma_star) @
    pynini.cdrewrite(pynini.cross("Á", "é"), slender_v + cons.plus, "", sigma_star)
)

# 4. Resolve Epenthesis for Autonomous '-tar' (Slender → -tear)
rule_epenthesis = pynini.cdrewrite(
    pynini.cross("T", "te"), slender_v + cons.plus, "ar", sigma_star)

# 5. Clean up boundary marker '+'
rule_cleanup = pynini.cdrewrite(pynini.cross("+", ""), "", "", sigma_star)

# The Comprehensive Surface Cascade
surface_cascade = (
    rule_A @ rule_I @ rule_Á @ rule_epenthesis @ rule_cleanup
).optimize()
```

#### The End-to-End Logic
When `first_conj_verbs.inflect()` is called:
1.  **Slot Selection**: Pynini selects the `+Aimid` slot for P1pl present.
2.  **Affixation**: It produces `mol+Aimid`.
3.  **Cascade Application**: The `surface_cascade` runs. 
    - `rule_A` sees `mol+Aimid` and ensures the archiphoneme `A` is broad (`a`).
    - `rule_I` looks inside the suffix, sees the `I` in `+Aimid`, and (following the vowel harmony of *mol*) converts it to `ai`, producing `molaimid`.
4.  **Final Surface**: The boundary `+` is removed, resulting in the valid Irish word **"molaimid"**.

This separation of **Morphotactics** (what suffix to add) from **Morphophonology** (how to spell it) is the hallmark of a professional-grade computational grammar.

## Automating Paradigm Hydration: The Hydrator Routine

Manually defining 112+ slots for a single part of speech is cumbersome and error-prone. In a production environment, we use a **Hydrator Routine** to ingest a textual representation of our reference matrix and automatically generate the Pynini `slots` list.

The following Python function illustrates how to convert a human-readable table string into the functional logic required by the `Paradigm` class.

```python
def hydrate_slots_from_table(table_data, stem_acceptor, category):
    """
    Ingests a textual matrix and returns a list of Pynini slots.
    Input Format: "Feature1=Val | Feature2=Val | ... | +Archiphoneme"
    """
    slots = []
    
    for line in table_data.strip().split("\n"):
        # Split line by pipe and strip whitespace
        parts = [p.strip() for p in line.split("|")]
        
        # The last part is the suffix; the rest are features
        suffix_marker = parts[-1]
        feature_strings = parts[:-1]
        
        # 1. Create the Affixation Transducer
        affix_fst = paradigms.suffix(suffix_marker, stem_acceptor)
        
        # 2. Create the FeatureVector
        # Filter out empty strings if the table has trailing pipes
        features_list = [f for f in feature_strings if "=" in f]
        f_vector = features.FeatureVector(category, *features_list)
        
        # 3. Append as a Pynini slot tuple
        slots.append((affix_fst, f_vector))
        
    return slots

# --- Example Usage ---

matrix_data = """
tense=pres | person=P1 | number=sg | +Im
tense=pres | person=P2 | number=sg | +Ann
tense=pres | person=P1 | number=pl | +Aimid
"""

# Hydrate the list
verb_slots = hydrate_slots_from_table(matrix_data, stem, verb_cat)

# Use in the Paradigm object
v_paradigm = paradigms.Paradigm(
    category=verb_cat,
    slots=verb_slots,
    rules=[surface_cascade]
)
```

### Why use a Hydrator?
1.  **Maintainability**: Changes to the linguistic model are made in the high-level matrix string, not deep within Python tuples.
2.  **Scalability**: The same routine can process matrices for Nouns, Adjectives, or Prepositions with no logic changes.
3.  **Auditability**: The source of truth for the grammar becomes a readable table that can be shared with non-developer linguists.

## The Paradigm Hydration Process (Pseudocode)

The following pseudocode outlines the exact logical process for "hydrating" a full paradigm for any Part of Speech (POS) in the Aithesc.io engine.

```text
PROCESS HydrateParadigm(POS_Name):
    1. ARRANGE Features:
       Define linguistic axes (Case, Number, Tense, person, etc.)
       Bundel into a Category(POS_Name)

    2. DEFINE Grammar Space Cells:
       Identify the cross-product of all Features (The "Grammatical Matrix").
       
    3. MAP Slots to Operands:
       FOR each distinct realization in the Grammatical Matrix:
           IF realization is common:
               Create a broad "Default" Slot (Underspecified FeatureVector)
           ELSE:
               Create an "Override" Slot (Fully specified FeatureVector)
           Bind Slot to an Affixation Transducer (e.g., paradigms.suffix("+Suffix"))

    4. DEFINE Morphophonological Cascade:
       Create FSTs for phonological shifts (Deletion, Palatalization, Harmony).
       Combine shifts into a single 'rules' cascade using composition (@).

    5. INSTANTIATE Paradigm:
       Pass Slots, Rules, and Dictionary Stems to paradigms.Paradigm().

    6. COMPILE Bidirectional Transducers:
       Access .inflector → Validates and constructs surface forms.
       Access .analyzer  → Mathematical inversion for parsing.

    RETURN Hydrated POS Object
```

This procedure ensures that the transition from abstract linguistic theory to a functional, bi-directional engine is both mathematically precise and computationally efficient.

### Example Analysis: Hydrating a 1st Conjugation Verb

To ground this procedure, consider the analysis required to hydrate the **1st person singular present** of the 1st conjugation verb *mol* (praise):

1.  **Linguistic Identification**: The engine identifies the semantic intent as `[lemma=mol] [tense=pres] [person=P1] [number=sg]`.
2.  **Cell Mapping**: In the "Grammatical Matrix," this cell is mapped to the abstract suffix `+Im` (an archiphoneme).
3.  **Affixation**: The `paradigms.suffix` operator creates the intermediate realization: `mol+Im`.
4.  **Phonotactic Resolution**: The `vowel_harmony` rule (Stage 4) scans the root *mol*. It detects the broad vowel *o* and the broad consonant *l*.
5.  **Surface Alignment**: The rule `rule_broad` (which targets the archiphoneme `I` followed by a broad environment) triggers a transformation, rewriting the string to the orthographically legal `molaim`.

By following this hydration cycle, the eGrammar ensures that every surface form—no matter how irregular—emerges from a structured, verifiable linguistic foundation.

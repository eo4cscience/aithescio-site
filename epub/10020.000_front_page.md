# Application Manual

## Welcome to the Aithesc.io Linguistic Modeling with FSTs.

This manual explains how to use the Finite State Transducer method for the modeling of computational grammars and for this project phonetics, morphology, and syntax of the Irish language. Mosy natural language processing: bithof the geenration and the understanding kinds, use Large Language Models (LLMs) nowadays.

There is a problem with the heavy reliance on these methods however because specifically for "under-documented" languages, these models are not very accurate. While these models are great for the English language, they are not very good for languages for which there are limited annotated corpora. The main problem with this lack of data is that the LLMs require vast amounts of training data and annotated corpora to be accurate and for Irish this is not the case. 

The solution lies in the combination of a much more rigorous description of the language together with these LLMs. This is what this project is about. The approach to take for providing a digital footptint to under-represented languages is to combine the strengths of both approaches. The formal eGrammars provide the structure and the LLMs provide the flexibility of natural language. 

**This Aithesc.io project is about the implementation of such a system for the Irish language**.

The purpose of the project is to provide a formal core model for the Irish language and wrap it in an LLM interface for text production and text understanding. 

The justification for this approach lies in the fact that this exactly how "we as humans" use language. In our comunication processes we don't use natural language in a purely probabilistic manner. We have learned a grammar, we have internalized the grammar of our mother tongue and any other language we are truly fluent in. The internalization of languages is a complex process in itself and very worthy of a study but for this project I will assume that the grammar is internalized as a result of osmosis (our mother tongue) or as a result of a formal study of the language (second language acquisition).

What makes working with the FST eGrammar core a possible solution component in the context of under-represented langauges is that it uses "very little information" to produce language models. Where it would take many terabytes of data to train an LLM to produce language, the FST eGrammar core uses only a few megabytes of data to produce language and to do so without errors or hallucinations This is because the FST eGrammar core is a formal model of the language, that takes very little space/memory and because it is rules based it is able to produce language in a deterministic manner. 

So are the FTSs the panacae? Unfortunately they are not. Just like LLMs are not the whole solution, so are the FSTs not the whole solution. The tension between Finite State Transducers (FSTs) and Large Language Models (LLMs) is essentially a battle between "perfect precision" and "graceful degradation."

While FSTs are the gold standard for rule-based linguistic tasks (like morphological analysis or phonology), their rigidity is their Achilles' heel. The core issue is that FSTs operate on a symbolic, discrete logic. If a word isn't in the lexicon or doesn't follow a predefined rule, the transducer simply fails to find a path through the state machine.

**Out-of-Vocabulary (OOV) Blindness**: Unlike an LLM, an FST doesn't "guess" based on context. If it encounters a neologism (e.g., "rizz") and it’s not in the state graph, the output is null or an error.

**The Embedding Advantage**: LLMs use vector embeddings, which place words in a high-dimensional continuous space. If an LLM sees an unknown word, it can still infer meaning based on sub-word tokens or the "geometric neighborhood" of surrounding words.

**Boolean vs. Probabilistic**: An FST is generally Boolean—a string either matches the grammar or it doesn't. LLMs are probabilistic, meaning they can find the "most likely" interpretation even if the input is messy.

This is why the industry is currently leaning toward Neuro-symbolic approaches. Instead of choosing one, developers use FSTs to enforce hard constraints (like ensuring a bank bot never says something illegal) while using LLM embeddings to handle the "fuzzy" natural language understanding of the user.

**It is along this line of though that Aithesc.io environment was built. The LLM human-interface is used to provide the "fuzzy" natural language understanding of the user and the FST eGrammar core is used to provide ground the ingested or intended meaning of the language in a formal manner. **
